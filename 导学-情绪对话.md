## 2. 重点亮点与学习顺序（先看这个）

### 亮点 1：SSE 流式响应的五层处理管线

**为什么重要**：这是项目的核心技术难点。从 HTTP 响应体到屏幕上逐字显示，中间经过 fetch → ReadableStream → TextDecoderStream → TransformStream（splitStream）→ JSON 解析 → 消息追加，五层管道每层职责清晰。

**先看哪些文件**：
1. `src/components/MarkdownPreview/models/index.ts` — 第 9-78 行 `createParser`，第 195-461 行各模型的 `chatFetch` 和 `transformStreamValue`
2. `src/components/MarkdownPreview/transform/index.ts` — `splitStream` TransformStream 实现
3. `src/store/business/index.ts` — `createAssistantWriterStylized` 组装管道
4. `src/views/chat.vue` — 第 100-123 行 `handleSend` 读取循环

**建议学习顺序**：从 `models/index.ts` 的 `chatFetch` 开始，理解请求怎么发；再看 `transform/index.ts` 理解数据怎么切；最后看 `chat.vue` 的读取循环理解数据怎么写入消息。

### 亮点 2：三层数据模型与 Pinia 状态隔离

**为什么重要**：Session / Message / Business 三个 Store 各司其职，通过 `sessionId` 关联，避免了单一大 Store 的维护噩梦。Message Store 内部用 `messagesBySession: Record<string, ChatMessage[]>` 按 sessionId 隔离，切换会话时零数据污染。

**先看哪些文件**：
1. `src/types/chat.d.ts` — 三个接口定义
2. `src/store/session/index.ts` — 会话 CRUD + localStorage 持久化
3. `src/store/message/index.ts` — 消息管理 + 流式状态（`isStreaming`、`activeReader`、`activeStreamingMessageId`）

**建议学习顺序**：先看 `chat.d.ts` 理解数据结构，再看 session store 理解会话生命周期，最后看 message store 理解流式状态如何与消息绑定。

### 亮点 3：流式中断与恢复机制

**为什么重要**：用户可以在 AI 生成过程中点击"停止"中断流式输出，中断后消息标记为 `failed`，支持重新生成。页面刷新时 `recoverIncompleteMessages()` 会将中断的流式消息恢复为 `failed` 状态，避免脏数据。

**先看哪些文件**：
1. `src/store/message/index.ts` — `abortActiveStream()`、`failMessage()`、`recoverIncompleteMessages()`
2. `src/views/chat.vue` — `handleStop` 和 `handleRetry` 处理

**建议学习顺序**：先理解 `abortActiveStream` 如何通过 `reader.cancel()` 中断流，再看 `recoverIncompleteMessages` 如何在组件挂载时清理脏数据。

### 亮点 4：多模型适配器模式

**为什么重要**：`modelMappingList` 用统一接口 `TypesModelLLM` 抽象了 7 种不同 LLM 的调用方式（OpenAI-compatible、Ollama、Spark 等），新增模型只需添加一个配置对象，无需改动业务代码。这是策略模式（Strategy Pattern）在前端的实际应用。

**先看哪些文件**：
1. `src/components/MarkdownPreview/models/index.ts` — 第 169-178 行 `TypesModelLLM` 接口定义

**建议学习顺序**：先看接口定义理解契约，再对比 Standard（mock）和 Kimi Moonshot（真实 API）两个实现，理解差异点在哪。

### 亮点 5：Vite 工程化配置

**为什么重要**：项目展示了 Vite 在实际项目中的高级用法：多代理规则、AutoImport 自动导入、UnoCSS 原子化 CSS、raw import 导入 .md 文件作为 mock 数据、动态 base 路径适配 GitHub Pages 部署。

## 7. 核心原理解析

### 7.1 SSE 流式传输

**问题**：AI 模型生成文本是逐 token 输出的，HTTP 一次性响应要等全部生成完才能返回，用户体验差。

**机制**：SSE（Server-Sent Events）基于 HTTP 长连接，服务端按 `data: {...}\n\n` 格式持续推送增量数据，客户端通过 `ReadableStream` 逐块读取。

**在本项目中的落点**：`models/index.ts` 中各模型的 `chatFetch` 设置 `stream: true`，服务端返回 SSE 流；`transform/index.ts` 的 `splitStream` 将原始字节流切分为独立的 JSON 事件；`chat.vue` 的读取循环将 `delta.content` 增量追加到消息。

### 7.2 TransformStream 管道

**问题**：原始字节流 → 可用文本 → 结构化事件，需要多步变换。

**机制**：Web Streams API 的 `TransformStream` 支持 `pipeThrough` 链式组合，每层只负责一种变换。

**在本项目中的落点**：`business/index.ts` 中 `response.body.pipeThrough(new TextDecoderStream()).pipeThrough(splitStream('\n'))`，两行代码完成字节→文本→事件的两级变换。

### 7.3 状态隔离与会话切换

**问题**：多会话场景下，切换会话时必须保证消息不串、流式状态不残留。

**机制**：Message Store 用 `messagesBySession: Record<string, ChatMessage[]>` 按 sessionId 分桶存储，`currentMessages` getter 通过 `activeSessionId` 过滤当前会话消息。

**在本项目中的落点**：`message/index.ts` 的 `currentMessages` getter 和 `addUserMessage` / `addAssistantMessage` 都接收 `sessionId` 参数，确保消息归属明确。

### 7.4 流式中断与异常恢复

**问题**：用户中断生成或页面刷新时，可能留下 `status: 'streaming'` 的半成品消息。

**机制**：`abortActiveStream()` 通过 `reader.cancel()` 中断 ReadableStream 读取，然后将消息标记为 `failed`；`recoverIncompleteMessages()` 在 Store 初始化时扫描所有消息，将残留的 `streaming` 状态修正为 `failed`。

**在本项目中的落点**：`message/index.ts` 第 100-120 行。

### 7.5 多模型适配器

**问题**：不同 LLM 服务商的 API 格式不同（OpenAI-compatible、Ollama 私有格式、Spark 私有格式），前端需要统一调用。

**机制**：策略模式 — 定义 `TypesModelLLM` 接口约束 `chatFetch` 和 `transformStreamValue` 两个方法签名，每种模型独立实现。

**在本项目中的落点**：`models/index.ts` 第 169-178 行接口定义 + 第 195-461 行 7 种实现。

## 8. 关键设计决策

### 决策 1：手动 localStorage 持久化 vs pinia-plugin-persistedstate

| 维度 | 手动 localStorage | pinia-plugin-persistedstate |
|------|-------------------|---------------------------|
| 灵活性 | 完全控制序列化时机和字段 | 自动但配置有限 |
| 包体积 | 零依赖 | 额外依赖 |
| 代码量 | 较多（每个 Store 都要写 load/persist） | 极少 |

**项目选择**：手动 localStorage。**风险**：忘记调用 `persistToStorage()` 会丢数据；`recoverIncompleteMessages` 是兜底方案。

### 决策 2：Vite Proxy vs 独立 Node 代理

| 维度 | Vite Proxy | 独立 Node 代理 |
|------|-----------|---------------|
| 部署 | 仅开发环境有效 | 开发+生产均可用 |
| API Key 安全 | 前端代码仍暴露 key（`import.meta.env.VITE_*`） | key 留在服务端 |
| 复杂度 | 零成本 | 需维护 Express/Koa 服务 |

**项目选择**：Vite Proxy（开发阶段）。**风险**：生产部署时 API Key 会暴露在前端 bundle 中，必须改为服务端代理。

### 决策 3：messagesBySession Record vs 单一 messages 数组 + filter

| 维度 | Record<sessionId, messages[]> | 单一数组 + filter |
|------|------------------------------|------------------|
| 查找效率 | O(1) 直接取 | O(n) 遍历过滤 |
| 切换会话 | 直接换 key | 重新 filter |
| 持久化 | 需序列化整个 Record | 序列化单一数组 |

**项目选择**：Record 分桶。**验证**：切换会话时无需重新计算，性能更优。

### 决策 4：SSE 解析格式自动检测

**备选**：硬编码 SSE 格式 / 配置化 / 自动检测。
**取舍**：`splitStream` 内部通过检查 `data:` 前缀和 `{` 开头自动判断格式，支持 SSE、JSON、plain text 三种模式。
**风险**：自动检测可能误判，但实际场景中格式是确定的，误判概率极低。

## 9. 量化与验证（含待测）

### 流式渲染性能

- **指标**：首 token 延迟（TTFT）、每秒 token 数（tokens/s）、滚动帧率
- **工具**：Chrome DevTools Performance 面板 + `performance.now()` 打点
- **基线**：mock 模式下 TTFT 应 < 100ms（无网络延迟）；真实 API 取决于模型响应速度
- **对比注意**：不同模型（DeepSeek vs Kimi vs Spark）的网络延迟差异大，需控制变量

### 消息持久化

- **指标**：1000 条消息的 localStorage 序列化/反序列化耗时
- **工具**：`performance.measure()` 包裹 `JSON.parse` / `JSON.stringify`
- **待测**：当前未做性能测试，大数据量下可能出现卡顿

### 内存占用

- **指标**：长会话（500+ 消息）下的内存峰值
- **工具**：Chrome DevTools Memory 面板 heap snapshot
- **待测**：`messagesBySession` Record 不会自动清理历史会话数据，可能存在内存泄漏风险

### 自动滚动

- **指标**：流式输出时的滚动流畅度（FPS）
- **工具**：Chrome DevTools Rendering > Frame Rendering Stats
- **待测**：`watch` 监听 last message content 变化频率过高时可能影响性能，可考虑 `requestAnimationFrame` 节流
