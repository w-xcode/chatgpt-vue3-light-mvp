# 面试背诵材料 — AI 情绪对话项目

> **使用说明**：这份材料模拟的是**真实对话的感觉**，不是标准答案。
> 面试时不要逐字背，而是把这些要点用自己的话串起来讲。
> 标记了 `🤔` 的地方是面试官可能会追问的方向，提前想好就行。

---

## 一、项目介绍（1 分钟版本）

**面试官**：简单介绍一下你的项目？

> 好的，我做的这个项目是一个 AI 情绪对话平台。简单来说就是用户可以跟 AI 聊天，聊完之后系统会自动分析这段对话的情绪状态——比如是开心、焦虑还是低落，然后把这些分析结果在看板页面用图表展示出来，用户可以看到自己情绪的变化趋势。
>
> 技术栈的话，前端是 Vue 3 加 Pinia 做状态管理，ECharts 做图表，AI 对话调的 DeepSeek 的 API，流式输出做的逐字渲染效果。
>
> 我在这个项目里主要做了三块内容：一块是流式对话，怎么把 AI 的回答一个字一个字显示出来，包括中断控制和异常恢复；第二块是情绪分析，因为 AI 返回的数据格式不太稳定，我做了好几层容错；第三块是数据看板，三种图表展示不同维度的情绪数据。

---

## 二、简历 bullet（6 条，面试前过一遍）

1. 基于 Fetch + ReadableStream + TransformStream 构建三层流式管道，解析 SSE 响应，实现逐字渲染和流式中断
2. 设计 Session / Message / EmotionReport 三层数据模型，Pinia 按 sessionId 隔离状态，持久化至 localStorage
3. LLM JSON 输出三层降级解析（直接解析 → markdown 提取 → 括号匹配），配合 clamp/default/cap 校验
4. Node 代理转发 API 请求，Key 存服务端环境变量，前端不接触任何密钥
5. ECharts Composable 封装，搭建趋势折线图、分布饼图、排行柱状图三类看板
6. AbortController 流式中断 + 启动时状态恢复，保障异常场景体验

## 三、核心问答

---

### 主题一：流式输出

---

#### Q1：流式输出怎么实现的？

**面试官可能这么问**：
- "AI 回答是逐字出现的，这个怎么做的？"
- "为什么不用 axios？"

> 这个的话，核心思路其实不复杂。就是浏览器发一个请求给服务端，但服务端不是等全部生成完了一次性返回，而是每生成一个字就立刻推过来，前端收到一个就显示一个。
>
> 具体实现上我用的是原生的 fetch，没用 axios。原因是 axios 底层是 XMLHttpRequest，它只能等整个响应回来之后才能拿到数据，不支持一块一块地读。但 fetch 不一样，它返回的 response.body 本身就是一个流，我可以一边接收一边处理。
>
> 大概流程就是，拿到数据块之后先解码成文字，然后按换行符把一整块拆成一行一行的，再从每一行里提取出 AI 生成的文字，追加到消息上。因为 Vue 是响应式的嘛，数据一变视图自动就更新了，用户看到的就是逐字出现的效果。
>
> 🤔 追问方向：管道链怎么工作的、SSE 协议格式、`data: [DONE]` 是什么

---

#### Q1 追问 1：管道链具体怎么工作的？

**面试官可能这么问**：
- "你说的管道链是什么意思？"
- "数据是怎么一步步处理的？"

> 嗯，管道链就是把几个处理步骤串起来，前一步的输出自动变成下一步的输入。
>
> 我这边分了三层：第一层是把二进制数据解码成文字，这一步是浏览器内置的；第二层是我自己写的一个处理器，它的作用是把一整块文字按换行符拆开，因为服务端推过来的数据可能一次包含好几行，也可能一行被拆成了好几次，所以这个处理器会把不完整的部分先存着，等下一次拼起来再输出；第三层就是从每一行里把有用的内容提取出来。
>
> 这样做的好处是每一层只管自己的事。比如以后我要换一种数据格式，只需要改第三层的解析逻辑，前面两层完全不用动。

---

#### Q1 追问 2：SSE 协议是什么？怎么知道数据传完了？

**面试官可能这么问**：
- "SSE 是什么？"
- "怎么判断 AI 回答完了？"

> SSE 其实就是一个基于 HTTP 的文本协议，服务端通过一个长连接持续往客户端推数据。每次推的数据之间用空行隔开，每块数据里有个 data 字段，里面就是实际的内容。
>
> 在 AI 对话的场景里，每个 data 字段里是一个 JSON，里面有个 content 字段，就是 AI 本次生成的几个字。
>
> 然后它有一个特殊信号叫 `data: [DONE]`，就是告诉客户端"我推完了，你可以关连接了"。我的代码检测到这个信号之后就退出读取循环，把消息状态从"生成中"改成"已完成"。

---

#### Q1 追问 3：用户想停止怎么办？网络断了呢？

**面试官可能这么问**：
- "用户点停止怎么处理？"
- "网络断开会怎样？"

> 停止的话我用的是 AbortController，这是浏览器原生的一个 API。发请求之前先创建一个控制器，把信号传给 fetch。用户点"停止"的时候调一下 abort，底层连接就断了，读取循环那边会收到一个中断信号。
>
> 关键是中断之后怎么处理。不管是用户主动停还是网络异常，都标记为"失败"，但已生成的部分内容会保留，用户能看到已经输出的片段，也可以点"重新生成"。
>
> 还有个恢复机制：应用每次启动的时候会扫一遍所有消息，如果发现有标记为"生成中"的——说明上次关闭的时候流还在跑——就把它们改成"失败"，避免界面上一直转圈。

---

#### Q1 追问 4：为什么不用 WebSocket？

**面试官可能这么问**：
- "SSE 和 WebSocket 有什么区别？"
- "为什么选 SSE 不选 WebSocket？"

> 这两个的核心区别是方向不同。SSE 是单向的，只能服务端往客户端推；WebSocket 是双向的，两边都能发。
>
> AI 对话这个场景，用户发一条消息之后就是等着 AI 回答，这是一个典型的单向推送，SSE 完全够用。WebSocket 的双向能力在这里用不上，反而增加了复杂度。
>
> 而且 SSE 基于 HTTP，天然就能走代理、CDN 这些基础设施，不需要额外适配。实现上 fetch 加流式处理也比 WebSocket 的 API 更自然。

### 主题二：状态管理

---

#### Q2：Pinia store 怎么设计的？

**面试官可能这么问**：
- "数据管理怎么做的？"
- "为什么分这么多 store？"

> 我把 store 按业务领域拆成了四个。SessionStore 管会话，就是左侧栏那些对话列表的增删改查；MessageStore 管消息，每个会话里的聊天记录；EmotionStore 管情绪报告，就是每次对话分析出来的结果；BusinessStore 管模型选择和流式管道的组装。
>
> 为什么要拆这么细呢，主要是单一职责的考虑。每个 store 只管自己那一块，改一个不会影响其他的。如果合成一个大 store，所有东西混在一起，代码会很难维护。
>
> 而且从依赖关系来看，会话和消息是可以独立用的，情绪报告依赖消息数据，模型选择被消息模块调用——这种依赖关系拆开之后一目了然，合成一个就看不出来了。

---

#### Q2 追问 1：多个对话的数据怎么隔离的？

**面试官可能这么问**：
- "切换对话的时候数据怎么分开？"
- "有没有遇到过数据串了？"

> 其实就是一个 Map，用会话 ID 做 key，每个会话的数据单独存一份。所有读写操作都得传会话 ID，这样不同会话之间就互不影响了。
>
> 不过开发的时候踩过一个坑——有一次切会话，上一个会话的 AI 还在回答呢，切过去之后新会话的界面上居然显示了旧会话正在生成的内容。后来排查发现是流式读取器没跟会话 ID 绑好，切换的时候旧的读取还在跑。加了个切换时先中断旧读取的逻辑就好了。
>
> 当时还挺奇怪的，数据明明隔离了怎么还会串，后来才反应过来，光在 store 层隔离没用，流式读取器那边也得跟着处理。

---

#### Q2 追问 2：数据怎么保存的？刷新页面还在吗？

**面试官可能这么问**：
- "数据持久化怎么做的？"
- "localStorage 有什么问题？"

>> 用的 localStorage，每个 store 启动的时候从里面读一次，数据变了就写回去。三个 store 各管各的 key，互相不牵扯。
>
> 好处就是简单嘛，不用装任何依赖，MVP 阶段够用了。但问题也挺明显的——容量只有 5MB，会话多了可能撑不住；而且它是同步的，数据量一大就会影响主线程，流式渲染的时候能感觉到卡；还有就是没有索引，所有数据都得全量读出来。
>
> 要改进的话可以换 IndexedDB，异步的、容量大、能建索引。或者干脆上后端数据库。

---

#### Q2 追问 3：为什么用 Pinia 不用 Vuex？

**面试官可能这么问**：
- "Pinia 和 Vuex 有什么区别？"

> 嗯，我用下来感受比较深的有几点。
>
> 首先就是 TypeScript 支持好，state 的类型能自己推导出来，写起来很顺，Vuex 那边光是类型声明就得写一大堆。
>
> 再就是 API 简单，Pinia 没有 mutations 那一层，actions 里直接改 state 就行，不用 commit 过去 commit 过来的，代码量少了不少。
>
> 还有模块化这块也舒服，每个 store 就是一个函数，用的时候导入就行。Vuex 的 modules 嵌套太深了，命名空间也绕，用着头疼。

### 主题三：情绪分析

---

#### Q3：LLM 返回的 JSON 格式不稳定怎么办？

**面试官可能这么问**：
- "AI 返回的数据格式不稳定怎么处理？"
- "JSON 解析出错怎么办？"

> 这个确实是我花精力比较多的地方。LLM 返回的 JSON 问题挺多的——有时候外面包了一层 markdown 代码块，有时候少了个字段，有时候数值类型不对，极端情况下 JSON 本身就是坏的。
>
> 我做了三层降级的策略：先直接解析，如果返回的是干净 JSON 这一步就成了；不行的话第二层，用正则把 markdown 代码块里的内容提取出来再解析；还不行的话第三层，从第一个大括号开始数，用计数器追踪嵌套，找到匹配的闭合大括号，截取出来再解析。
>
> 解析完之后还有字段级的校验，比如数值超范围就裁剪，数组太长就截断，枚举值不匹配就给默认值。总之就是确保不管 AI 返回什么，最终写进 store 的数据一定是合法的。

---

#### Q3 追问 1：什么时候做情绪分析？会影响对话速度吗？

**面试官可能这么问**：
- "情绪分析是同步还是异步的？"
- "会不会让对话变慢？"

> 是异步的，AI 回答完之后才触发，不阻塞主流程。
>
> 具体就是 AI 回答生成完了，我拿这段回答去调一次 AI，让它按 JSON 格式返回情绪标签、评分这些。因为是异步的，用户感觉不到延迟，对话结束后一两秒标签就出来了。
>
> 如果分析失败了也不影响对话，只是不显示标签而已。设计上的考虑是，如果等分析完了再显示回答，用户要多等一两秒，体验不好。异步的话标签出现有轻微延迟，但用户基本感知不到。

---

#### Q3 追问 2：情绪分析的 prompt 怎么写的？

**面试官可能这么问**：
- "你给 AI 的提示词怎么设计的？"
- "怎么保证它按格式返回？"

> prompt 分两部分。system prompt 定义角色和输出格式——要求严格按 JSON 返回，包含哪几个字段，每个字段什么类型、什么范围都写清楚了。user prompt 就是把 AI 的回答原文传进去。
>
> 保证格式这块，我的策略是"不完全信任"。就算 prompt 里写了必须返回 JSON，AI 有时候还是会返回奇怪的格式，所以三层降级解析就是兜底。prompt 里我加了一句"只返回 JSON，不要有其他文字"，这个能明显降低出问题的概率。
>
> 情绪分析用的是同一个 AI 模型，没用更小的，因为情绪分析需要理解对话语义，小模型不太靠谱。

---

#### Q3 追问 3：clamp、cap 这些校验怎么想到的？

**面试官可能这么问**：
- "这些边界处理怎么考虑的？"
- "为什么要裁剪？"

> 都是从实际跑出来的 bug 里总结的。比如评分我要求返回 0 到 10 的整数，但它经常给我 7.5 这种小数，甚至偶尔来个 12 或者负数，不裁剪不行。
>
> 标签数组也是，AI 有时候一口气返回十几个，前端根本展示不下，截断到合理的数量。
>
> 风险等级我定义了三个值，但 AI 有时候返回中文，有时候返回英文，有时候大小写都不一样，所以做了模糊匹配，匹配不上就给个安全的默认值。
>
> 说白了就是不能信任外部输入，每一层都得有兜底。

### 主题四：数据看板

---

#### Q4：ECharts 看板怎么做的？

**面试官可能这么问**：
- "图表怎么实现的？"
- "Composable 怎么封装的？"

> 看板做了三个图表：情绪趋势的折线图、情感分布的饼图、关键词排行的柱状图。
>
> 我封装了一个通用的 hook，把 ECharts 的生命周期和 Vue 绑在一起。传一个 DOM 容器进去，初始化、更新、销毁都自动化了。用的时候组件只需要准备好数据，剩下的 hook 全处理。
>
> 三个图表都用同一个 hook，代码量少很多。而且因为数据是响应式的，store 里的情绪报告变了，图表自动就更新了，不用手动调任何方法。

---

#### Q4 追问 1：三个图表分别展示什么？数据怎么联动？

**面试官可能这么问**：
- "图表里展示的是什么数据？"
- "数据怎么从 store 流到图表？"

> 折线图是情绪评分随时间的变化，X 轴时间，Y 轴 0 到 10 分，能看出情绪的起伏。饼图是情绪标签的分布，比如"开心"出现了多少次、"焦虑"多少次，看整体的情感倾向。柱状图是关键词的 Top 10 排行，看哪些话题被提到最多。
>
> 数据流的话，store 里有个 getter 合并所有会话的情绪报告，页面里用计算属性从这个数组派生出图表需要的数据。因为是响应式的嘛，store 数据变了 → getter 重算 → 计算属性重算 → 图表自动更新，整个链路是自动流转的。

---

#### Q4 追问 2：ECharts 在 Vue 里有什么坑？

**面试官可能这么问**：
- "用 ECharts 遇到过什么问题？"

> 最大的坑是初始化时机。ECharts 需要一个已经有尺寸的 DOM 容器才能初始化，如果容器还没渲染完或者尺寸是 0，图表会显示不出来。我的方案是懒初始化，第一次设置配置项的时候才真正去 init，保证容器一定已经就绪了。
>
> 第二个是内存泄漏。组件卸载的时候如果不销毁 ECharts 实例，内存会一直涨，resize 监听器也不会自动移除。所以在组件卸载时做了实例销毁和事件清理。
>
> 不过我这边数据变化频率不高，只有新生成情绪报告的时候图表才更新，性能上没遇到太大问题。

### 主题五：API 安全

---

#### Q5：API Key 安全怎么处理的？

**面试官可能这么问**：
- "API Key 怎么保管的？"
- "为什么不直接前端调 API？"

> 核心原则就是前端永远不碰密钥。
>
> 我搭了一个 Node 代理服务，前端请求先发到自己的代理，代理把 API Key 注进去再转发给 DeepSeek。整个链路是：浏览器 → 我的代理 → DeepSeek API。
>
> Key 存在环境变量里，没带 VITE_ 前缀，所以不会被打包进前端代码。.gitignore 也排除了 .env 文件，只提交了一个模板。
>
> 这样即使有人看前端代码或者打开网络面板，也看不到任何密钥。代理层代码量很小，核心逻辑大概 30 来行。

---

#### Q5 追问 1：生产环境怎么搞？

**面试官可能这么问**：
- "上线的话代理怎么部署？"

> 思路是一样的，载体不同而已。
>
> 比较常见的方案是 Nginx 反向代理，配置一个 location 匹配 API 路径，用 proxy_pass 转发，顺便注入 Key。性能好，配置也成熟。
>
> 或者用 Serverless Function，比如 Vercel Edge Function，写一个很薄的代理层。好处是不用管服务器，自动扩缩容。
>
> 不管哪种方案，核心原则不变——Key 只在服务端，前端只跟自己的代理通信。

---

#### Q5 追问 2：为什么不直接用 Vite proxy？

**面试官可能这么问**：
- "Vite 不是自带 proxy 吗？"

> Vite proxy 只在开发的时候跑，打包之后的产物不包含 Vite 的开发服务器。如果用它注入 Key，生产环境还得另外搞一套代理，开发和生产走两套架构，维护起来麻烦。
>
> 单独起 Node 服务的话，开发和生产的代理代码是同一套，架构统一。而且职责也清晰——Vite 只管构建和热更新，代理只管转发和注入 Key，各干各的。

### 主题六：路由

---

#### Q6：路由和权限怎么做的？

**面试官可能这么问**：
- "路由怎么配的？"
- "有没有权限控制？"

> 路由很简单，/chat 是对话页，/dashboard 是看板页，/ 重定向到 /chat，其他路径 404。
>
> 目前是 MVP 阶段，没有真正的登录和权限校验，只做了进度条的控制。
>
> 如果要扩展的话，思路是在路由守卫里检查 token：没有 token 且目标页面需要登录就跳登录页；有 token 就解析角色信息，跟路由声明的权限做比对。Vue Router 的 beforeEach 守卫每次路由切换都会触发，是做认证逻辑的标准位置。

---

#### Q6 追问 1：如果加管理员角色怎么做？

**面试官可能这么问**：
- "多用户怎么搞？"
- "管理员能看所有人数据怎么做？"

> 路由和数据两层一起做。路由层在看板页面限制只有 admin 能访问，普通用户看不到入口，手动输 URL 也会被拦住。
>
> 数据层的话，现在是纯前端没有用户概念，要支持多用户就得上后端数据库了，每个用户的数据跟用户 ID 绑定。管理员的接口查所有人的，普通用户查自己的。前端调接口的时候带 token，后端根据 token 判断身份和权限，返回对应范围的数据。
>
> 看板组件本身不用改，它只是渲染数据，区别就是数据源的范围变了。

#### Q7：Markdown 渲染怎么做的？

**面试官可能这么问**：
- "AI 回答里的代码高亮怎么做的？"
- "公式和图表怎么渲染？"

> 用 markdown-it 做核心引擎，然后插件扩展。代码高亮用的 highlight.js，公式用的 KaTeX，Mermaid 图表用的 mermaid-js。
>
> 代码块外面还包了一层容器，加了语言标签和一键复制按钮。这些插件是按需加载的，只有遇到对应格式才触发，不会一上来全部初始化。

---

#### Q7 追问 1：DeepSeek 的推理过程怎么展示？

**面试官可能这么问**：
- "思考过程和最终回答怎么区分？"
- "think 标签怎么处理的？"

> DeepSeek 的推理模型会同时输出思考过程和最终回答两部分内容。我把思考过程包在特殊标签里，渲染的时候识别出来做成一个可折叠的区域，默认收起来，用户想看可以展开。
>
> 这样推理过程不会干扰正常阅读，但想了解 AI 怎么想的也能看到。流式输出的时候，用户会先看到思考过程逐字出现，想完了最终回答才开始出来，体验跟 DeepSeek 官方是一致的。

---

#### Q7 追问 2：输入框怎么做的？

**面试官可能这么问**：
- "Enter 和 Ctrl+Enter 有什么区别？"
- "发送逻辑怎么设计的？"

> 输入框是一个 textarea，支持 Enter 直接发送和 Ctrl+Enter 换行。这个是考虑到不同用户习惯，有人习惯回车就发，有人习惯 Ctrl+回车发。
>
> 发送前会检查两个条件：输入框有内容，而且当前没有正在生成的回答。两个都满足才发，发完清空输入框。
>
> 还做了快捷标签功能，预设一些常用提示词，点一下就填进去。生成过程中会显示停止按钮。

## 六、踩坑故事（被问"遇到什么坑"时讲）

**坑 1：切换会话数据串了**
> 有一次切会话的时候发现新会话里显示的是上一个会话正在生成的内容，debug 了半天发现是流式读取器没跟会话 ID 绑好，切换的时候旧的读取还在跑。加了个切换时中断的逻辑就好了。

**坑 2：ECharts 图表显示不出来**
> 刚开始在组件挂载的时候就初始化图表，结果容器还没渲染完，尺寸是 0，图表一片空白。后来改成懒初始化，第一次设置数据的时候才真正创建图表，问题就解决了。

**坑 3：AI 返回的 JSON 解析失败**
> 第一次做情绪分析的时候直接 JSON.parse，结果报错。打印出来一看，AI 把 JSON 包在了 markdown 代码块里。后来加了正则提取代码块内容的逻辑，才解决了这个问题。这也是我做三层降级解析的起因。
