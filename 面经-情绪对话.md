# 面经 — AI 情绪对话与数据分析平台

## 1. 项目简介（简历可用）

基于 Vue 3 + Pinia + Vite 开发的 AI 多模型对话平台，核心实现 SSE 流式多轮对话、会话管理、Markdown 富文本渲染（KaTeX / Mermaid / 代码高亮），支持 DeepSeek / Kimi / Spark / SiliconFlow / Ollama 等 7 种 LLM 的统一接入与切换。

## 2. 简历 Bullet（4～6 条）

- 基于 fetch + ReadableStream + TransformStream 实现 SSE 流式响应解析管线，将 Kimi/DeepSeek 等 API 的 delta token 增量写入消息，支持逐字渲染、生成中断和重新生成，首 token 延迟（待测）。
- 设计 Session / Message 两层数据模型，使用 Pinia 按 sessionId 隔离消息存储，配合 localStorage 手动持久化实现会话恢复，切换会话零数据污染。
- 通过 Vite dev server proxy 转发 4 家 LLM 服务商 API 请求，统一 OpenAI-compatible 接口配置，新增模型仅需添加一个配置对象，无需改动业务代码。
- 实现流式中断与异常恢复机制：`reader.cancel()` 中断读取 + 消息标记 failed + 页面刷新时自动修正残留 streaming 状态。
- 集成 markdown-it 插件链（KaTeX 数学公式、Mermaid 图表、highlight.js 代码高亮），实现 AI 回复的富文本实时渲染。
- 基于 Vue Router + NProgress 实现路由导航守卫，通过 Vite AutoImport + UnoCSS + Naive UI 搭建组件化工程架构。

### 主题一：SSE 流式通信（核心亮点）

#### Q1：你是怎么实现 AI 对话的流式输出的？

面试官你好，我来详细讲一下这个项目的流式输出实现。当时做这个功能的背景是，AI 模型生成文本是逐 token 输出的，如果用普通的 HTTP 请求，用户需要等模型全部生成完才能看到结果，体验非常差，尤其是长文本场景下可能要等十几秒。所以我们需要一种机制，让前端能够边接收边渲染，实现"打字机"效果。

技术选型上，我选择了基于 fetch API 的 ReadableStream 来接收 SSE 流。具体来说，当用户发送消息时，前端调用 `fetch()` 发起请求，设置 `stream: true`，服务端返回的不是一次性 JSON，而是一个持续推送的字节流。拿到 `response.body` 这个 ReadableStream 后，我通过 `pipeThrough` 串联了两个 TransformStream：第一个是浏览器原生的 `TextDecoderStream`，负责把 Uint8Array 字节解码成字符串；第二个是我自己实现的 `splitStream('\n')`，负责把连续的文本按换行符切分成独立的事件行。

在 `splitStream` 的实现里，我做了一个格式自动检测逻辑：检查每行是否以 `data:` 开头（SSE 格式）还是以 `{` 开头（JSON 格式），然后分别走不同的解析路径。对于 SSE 格式，去掉 `data: ` 前缀后尝试 JSON parse，提取 `choices[0].delta.content` 字段。对于 JSON 格式，用子串扫描的方式找到完整的 JSON 对象。解析出文本片段后，在 `chat.vue` 的读取循环里，通过 `messageStore.appendToMessage(assistantMsg.id, text)` 增量追加到消息的 content 字段，Vue 的响应式系统会自动触发视图更新，实现逐字渲染。

最终效果是，用户点击发送后几乎立即看到 AI 开始"打字"，每收到一个 token 就渲染一个，体验非常流畅。

#### Q2：SSE 解析过程中遇到过什么坑？

关于 SSE 解析的坑，我遇到过几个比较典型的问题。第一个是缓冲区拼接问题：TCP 传输是基于字节流的，一个 SSE 事件可能被拆成多个 TCP 包到达，也可能多个事件合并成一个包。我的 `splitStream` 实现里用了一个 `lastLineTail` 缓冲变量，每次读取新 chunk 时先和上次的尾部拼接，再按换行符切分，确保不会在事件中间截断。这个细节如果处理不好，会出现 JSON 解析失败或者丢数据的情况。

第二个坑是不同 LLM 服务商的 SSE 格式不完全一致。DeepSeek 会在事件之间插入 `: keep-alive` 这样的注释行做心跳保活，如果不处理就会导致 JSON parse 报错。我的 `createParser` 里专门加了对冒号开头行的过滤逻辑。另外 DeepSeek 的 reasoning model（R1）有一个特殊的 `reasoning_content` 字段，和普通的 `content` 是分开传输的，我用 `createStreamThinkTransformer` 做了一个状态机，检测到 `reasoning_content` 时用 `<think>` 标签包裹，切换到 `content` 时自动闭合。

第三个是 `[DONE]` 终止信号的处理。不同服务商的终止方式不同，有的发 `data: [DONE]`，有的直接关闭连接。我在解析器里同时处理了这两种情况，确保流能正确结束。

#### Q3：流式输出过程中用户点击"停止"是怎么处理的？

这个功能的实现分三个层面。首先是中断读取：当用户点击停止按钮时，我调用 `messageStore.abortActiveStream()`，这个方法会拿到当前活跃的 `ReadableStreamDefaultReader`，调用它的 `cancel()` 方法。`reader.cancel()` 会通知浏览器停止读取流，底层会中断 HTTP 连接。

然后是状态更新：中断后，我把当前正在生成的助手消息的 status 从 `streaming` 改为 `failed`，同时清空 Store 里的 `activeReader`、`activeStreamingMessageId` 和 `isStreaming` 状态。这样 UI 上会显示"生成失败"的提示，而不是一个永远转圈的加载状态。

最后是兜底恢复：考虑到用户可能直接关闭浏览器标签页而不是点停止按钮，我在 Message Store 初始化时会调用 `recoverIncompleteMessages()`，遍历所有会话的所有消息，把残留的 `streaming` 状态强制修正为 `failed`。这个方法在 `onMounted` 的时候执行，确保每次打开应用数据都是干净的。

用户看到 failed 状态的消息后，可以点击重新生成，这时候会用相同的用户消息重新发起一次流式请求。整个链路是：停止 → 中断流 → 标记失败 → 重新生成 → 新建流。

### 主题二：状态管理与数据模型

#### Q4：为什么把 Session 和 Message 分成两个 Store？

这个设计是基于职责分离的考虑。Session Store 管理会话的元数据：id、标题、创建时间、更新时间，它的操作是低频的——创建、删除、重命名、切换。Message Store 管理每个会话里的具体消息和流式状态，它的操作是高频的——每收到一个 token 就要调用 `appendToMessage`。

如果合到一个 Store 里，每次追加 token 都要触发整个 Store 的响应式更新，包括不相关的会话列表排序、标题生成等逻辑，性能浪费。分开之后，Message Store 的 `appendToMessage` 只修改 `messagesBySession[sessionId]` 里某条消息的 content，影响范围最小。

另外，流式状态（`isStreaming`、`activeReader`、`activeStreamingMessageId`）天然属于消息域，放在 Message Store 里语义更清晰。如果放在 Session Store，就要在会话对象上挂一堆和"会话"概念无关的字段，模型会变得很臃肿。

两个 Store 之间通过 `sessionId` 关联：Session Store 的 `activeSessionId` 决定了 Message Store 的 `currentMessages` getter 返回哪些消息。这种松耦合的设计让两个 Store 可以独立演化，比如未来要给 Session 加标签分组功能，完全不影响 Message Store。

#### Q5：messagesBySession 用 Record 而不是单一数组加 filter，有什么好处？

用 `Record<string, ChatMessage[]>` 做分桶存储，最大的好处是切换会话时的时间复杂度是 O(1)。`currentMessages` getter 直接通过 `messagesBySession[activeSessionId]` 取值，不需要遍历整个消息数组。如果用单一数组加 filter，每次切换会话都要 O(n) 遍历，当消息量大的时候（比如积累了几百个会话、上万条消息），切换会有明显卡顿。

还有一个好处是数据隔离。每个会话的消息在内存上是独立的数组引用，修改一个会话的消息不会触发其他会话的响应式更新。如果用单一数组，修改任意一条消息都会触发整个数组的依赖收集和 diff 计算。

持久化方面，Record 结构直接 `JSON.stringify` 写入 localStorage，读取时 `JSON.parse` 一次性恢复，代码非常简洁。缺点是无法单独持久化某个会话的消息，但当前场景下这个缺点不重要，因为消息量还没大到需要分片存储的程度。

#### Q6：localStorage 持久化有什么局限性？你会怎么改进？

localStorage 的局限性主要有三个。第一是容量限制，一般 5MB 左右，当消息量很大（比如几百个会话、每个会话几十条消息，每条消息包含长文本的 Markdown）时，可能超出限制导致 `setItem` 抛异常。第二是同步 API，`JSON.stringify` 大对象时会阻塞主线程，消息多了之后在 `persistToStorage` 调用时可能出现可感知的卡顿。第三是没有索引和查询能力，每次都要全量序列化和反序列化。

改进方案的话，短期可以加一个 try-catch 包裹 localStorage 写入，捕获 QuotaExceededError 时做 LRU 淘汰——删掉最久没访问的会话。中期可以迁移到 IndexedDB，用 idb-keyval 这类轻量库，容量大得多（几百 MB），而且支持异步读写不阻塞主线程。长期的话，如果要做多端同步，可以考虑接后端数据库，localStorage 只做缓存层。

当前项目里还有一个优化点：`persistToStorage` 是在每次 `completeMessage` 时调用的，也就是只有流式结束才持久化，中间的增量更新不写盘。这减少了写入频率，但如果页面在流式过程中崩溃，中间数据会丢失。可以考虑加一个防抖的自动保存，比如每 5 秒保存一次。

### 主题三：多模型适配与 API 设计

#### Q7：你是怎么支持多种 LLM 的？新增一种模型需要改多少代码？

我用了一个类似策略模式的设计。核心是在 `src/components/MarkdownPreview/models/index.ts` 里定义了一个 `TypesModelLLM` 接口，约束了两个方法：`chatFetch` 负责发起 API 请求，`transformStreamValue` 负责把原始流数据转换成统一的 `{ content }` 或 `{ done }` 格式。

每种 LLM 实现这个接口。比如 Standard（mock 模型）的 `chatFetch` 是创建一个从本地 markdown 文件读取的 ReadableStream，不需要真实网络请求；Kimi Moonshot 的 `chatFetch` 是发一个 POST 到 `/moonshot/v1/chat/completions`，body 是 OpenAI-compatible 格式；Ollama 的 `chatFetch` 则是发到 `localhost:11434/api/chat`，body 格式和 OpenAI 不同，需要单独处理。

新增一种模型的话，只需要在 `modelMappingList` 数组里加一个对象，实现 `chatFetch` 和 `transformStreamValue` 两个方法，然后在 Vite config 里加一条 proxy 规则（如果需要跨域代理的话）。业务代码完全不用改，因为 `business/index.ts` 的 `createAssistantWriterStylized` 是通过 `currentModelItem` getter 动态获取当前模型的，解耦很彻底。

目前项目支持 7 种模型：Standard（mock）、DeepSeek-V3、DeepSeek-R1（带 reasoning 内容）、Ollama 3、Spark、SiliconFlow、Kimi Moonshot。其中 DeepSeek-R1 最特殊，因为它有 `reasoning_content` 字段，我在 `transformStreamValue` 里做了一个状态机来处理 thinking 和 content 的切换。

#### Q8：Vite Proxy 在生产环境能用吗？你怎么解决 API Key 安全问题？

Vite Proxy 只在开发环境生效，它是 Vite dev server 的一部分，`vite build` 之后产物是纯静态文件，没有 Node 服务来处理代理。所以生产环境直接把 API Key 放在前端代码里，Key 会暴露在 JS bundle 中，任何人打开 DevTools 都能看到，这是严重的安全问题。

当前项目的处理方式是：在 GitHub Pages 部署模式下（`isGithubDeployed` 为 true），直接禁用所有非 mock 模型，用户只能用 Standard 模式体验 demo，不发真实 API 请求。这是一种权宜之计，适合演示但不适合生产。

正确的解决方案是搭建一个独立的 Node.js 代理服务。前端请求先发到自己的代理服务，代理服务在服务端环境变量里读取 API Key，然后转发请求到 LLM 服务商。这样 Key 永远不会出现在前端代码中。具体实现可以用 Express 或 Koa，加一个 `/api/chat` 路由，接收前端的消息数组，附上服务端的 Key，转发到对应的 LLM API。还可以在代理层加限流、鉴重、日志等功能。

另外，如果要支持多用户，代理层还可以做用户鉴权（JWT 或 session），每个用户有自己的配额，避免 Key 被滥用。这是从 MVP 到生产级必须补的架构短板。

### 主题四：Markdown 渲染与 UI

#### Q9：AI 回复里的数学公式、图表、代码高亮是怎么实现的？

我用 markdown-it 作为 Markdown 解析核心，通过插件机制扩展了三种富文本能力。第一是 KaTeX 数学公式：在 markdown-it 的渲染规则里拦截行内公式（`$...$`）和块级公式（`$$...$$`），交给 KaTeX 渲染成 HTML。第二是 Mermaid 图表：检测代码块的语言标记如果是 `mermaid`，就调用 Mermaid.js 的 `render` 方法生成 SVG。第三是代码高亮：用 highlight.js 对代码块做语法高亮，支持 Vue、JavaScript、Python 等常用语言，还自定义了 Mermaid 语言定义以支持高亮。

为了安全，渲染后的 HTML 会经过 DOMPurify 做 XSS 过滤，防止 AI 回复中注入恶意脚本。这是一个容易被忽略但很重要的安全点——AI 模型的输出是不可信的，理论上可能生成包含 `<script>` 标签的内容。

代码块还有一个交互细节：右上角有复制按钮。我在 `preWrapper.ts` 插件里给每个 `<pre>` 块包了一层容器，加了一个按钮，点击时调用 `navigator.clipboard.writeText` 复制代码内容。

流式渲染的时候有个体验问题：KaTeX 和 Mermaid 需要完整的公式/图表语法才能渲染，但流式输出是逐字符到达的，中间状态可能语法不完整。我的处理方式是在流式过程中先用纯文本显示，等消息 status 变成 completed 后再触发完整的 Markdown 渲染。这样避免了中间状态的渲染错误。

#### Q10：消息列表的自动滚动是怎么做的？

自动滚动的实现用了 Vue 3 的 `watch` API，监听两个数据源。第一个是消息数量变化：当有新消息添加时（用户发消息或 AI 开始回复），触发 `scrollToBottom`。第二个是流式内容增长：当最后一条消息的 status 为 `streaming` 时，watch 它的 content 字段，每次内容追加都触发滚动。

`scrollToBottom` 的实现是拿到消息列表容器的 DOM 元素，设置 `scrollTop = scrollHeight`，强制滚到底部。这里用 `nextTick` 确保 DOM 已经更新后再滚动，否则新的消息内容还没渲染出来，滚动距离算不对。

有一个用户体验的考量：如果用户手动往上翻看了历史消息，这时候不应该强制把他拉到底部。可以在 watch 里加一个判断——只有当用户当前已经在底部附近（比如距离底部 50px 以内）时才自动滚动，否则保持当前位置。当前实现没有加这个判断，是一个可以优化的点。

### 主题五：工程化与工具链

#### Q11：Vite 的 AutoImport 是怎么配的？解决了什么问题？

AutoImport 解决的是反复手动 import 的问题。我在 `vite.config.ts` 的 `AutoImport` 插件里配了几个自动导入源：Vue 的 `ref`、`computed`、`watch` 等 API；Vue Router 的 `useRouter`、`useRoute`；VueUse 的 `useMagicKeys`、`useClipboard` 等工具函数；Naive UI 的 `useMessage`、`useDialog` 等钩子；还有 `uuid` 的 `uuidv4` 和 `lodash-es` 的常用函数。

配完之后，代码里直接用 `ref()`、`useRouter()` 这些，不需要写 import 语句。插件会自动扫描源码，在构建时插入对应的 import，同时生成 `auto-imports.d.ts` 给 TypeScript 提供类型提示。开发体验提升很大，少了大量样板代码。

还有一个配套的 `Components` 插件，自动注册 Naive UI 的组件和 Iconify 图标，用 `<n-button>`、`<n-input>` 这些标签不需要手动注册。两个插件配合，代码里基本不用写任何 import 和 component 注册。

注意点是 `.eslintrc-auto-import.json` 这个文件需要提交到仓库，否则 ESLint 会报未定义变量的错误。这个文件是 AutoImport 插件自动生成的，包含所有自动导入的全局变量声明。

#### Q12：UnoCSS 在项目里是怎么用的？和 Tailwind 有什么区别？

UnoCSS 是一个原子化 CSS 引擎，和 Tailwind 类似但更轻量。我在项目里用它做样式，比如 `class="flex items-center gap-2 p-4"` 这种 utility class 直接写在模板里，不需要写 CSS 文件。

和 Tailwind 的主要区别是：UnoCSS 是按需生成的，只生成你实际用到的 CSS 规则，不像 Tailwind 会生成一个很大的预设样式表。它的引擎是用正则匹配实现的，扩展性很强——可以自定义规则、预设和 transformers。我在项目里用的是默认预设（`presetUno`），支持 Tailwind / Windi CSS / Bootstrap 的大部分语法。

在 `vite.config.ts` 里配置了 `UnoCSS()` 插件，它会在构建时扫描所有模板文件，提取用到的 class，生成对应的 CSS。开发模式下是即时生成的，HMR 速度很快。产物体积方面，因为是按需生成，比 Tailwind 的完整 CSS 小得多。

### 主题六：组件设计与交互

#### Q13：ChatInput 组件是怎么做快捷键适配的？

ChatInput 需要支持快捷键发送消息，但 macOS 和 Windows 的修饰键不同——macOS 用 Command+Enter，Windows 用 Ctrl+Enter。我的实现分两步：第一步用 `ua-parser-js` 检测当前操作系统是否是 macOS，根据结果动态设置 placeholder 文本（显示"Command"还是"Ctrl"）。第二步用 VueUse 的 `useMagicKeys` 监听键盘事件，分别注册 `Meta+Enter` 和 `Ctrl+Enter` 两个快捷键组合。

`useMagicKeys` 的好处是它自动处理了修饰键的平台差异——在 macOS 上 `Meta` 就是 Command 键，在 Windows 上就是 Win 键。但实际测试发现，Windows 上用户习惯用 Ctrl+Enter 而不是 Win+Enter，所以我单独监听了 `Ctrl+Enter`。两个快捷键用 `logicOr` 组合，任意一个触发就执行发送。

输入框还支持一个细节：当内容为空时，发送按钮是禁用的，防止用户发空消息。这是用 computed 属性 `canSend` 做的判断，绑到按钮的 `disabled` 属性上。

#### Q14：SideBarItem 的 hover 交互和删除确认是怎么做的？

SideBarItem 需要在鼠标 hover 时显示编辑和删除图标，未 hover 时隐藏。我的实现是用一个 `hovered` ref 状态，在 `@mouseenter` 和 `@mouseleave` 事件里切换。图标用 `v-show="hovered || active"` 控制显示——hover 状态或当前激活的会话都显示图标。

删除操作用 Naive UI 的 `n-popconfirm` 组件做二次确认。用户点击删除图标后弹出一个小气泡，显示"确认删除？"和确认/取消按钮，点击确认才执行删除。这比 `window.confirm` 体验好，因为它是在页面内渲染的，不会阻塞主线程，样式也和 Naive UI 主题一致。

编辑标题是 inline 编辑模式：点击编辑图标后，标题文本替换成一个 `n-input` 输入框，用户输入新标题后，按 Enter 或失焦（blur）触发保存。失焦用 `@blur` 事件监听，Enter 用 `@keyup.enter` 监听，两种方式都会调用 `sessionStore.renameSession` 并退出编辑模式。

#### Q15：消息气泡是怎么区分用户和 AI 的？

消息气泡的布局通过 `ChatMessage` 组件的 `role` prop 控制。用户消息右对齐，背景色用 Naive UI 的主色（紫色），文字白色；AI 消息左对齐，背景色用浅灰色，文字深色。这是在组件模板里用 `:class` 动态绑定实现的。

AI 消息的内容通过 `v-html` 渲染，调用 `renderMarkdownText()` 把 Markdown 文本转成 HTML。这个函数内部用 markdown-it 解析，经过 KaTeX、Mermaid、highlight.js 三个插件处理，最后用 DOMPurify 过滤 XSS。

失败状态的消息（status === 'failed'）会显示一个警告图标和"生成失败"文字，替代正常的 Markdown 内容。这是用 `v-if` 做的条件渲染。

流式状态的消息（status === 'streaming'）没有特殊的加载动画，内容就是当前已收到的文本。这是一个体验优化点——可以加一个闪烁的光标效果（cursor blink），让用户更直观地知道 AI 正在"打字"。

### 主题七：异常处理与边界情况

#### Q16：流式过程中网络断开怎么办？

网络断开时，`reader.read()` 会抛异常或者返回 `{ done: true }`。我在 `chat.vue` 的读取循环里用 try-catch 包裹了整个 while 循环，捕获异常后调用 `messageStore.failMessage(assistantMsg.id)` 把消息标记为 failed。这样用户看到的是"生成失败"提示，而不是页面崩溃。

如果是部分生成后断开，已生成的内容会保留——因为 `appendToMessage` 是增量追加的，之前追加的文本不会回滚。用户重新生成时，会创建一条新的助手消息，而不是在原来的基础上继续。这是一个设计选择：重新生成比断点续传简单可靠，代价是丢失已生成的部分内容。

#### Q17：localStorage 数据被手动篡改或损坏怎么办？

我在 `loadFromStorage` 里加了 try-catch，`JSON.parse` 失败时返回空数组，不会让应用崩溃。这是一个防御性编程的实践——localStorage 是用户可操作的，不能假设它的内容一定是合法的。

更进一步的话，可以在序列化时加一个版本号字段，反序列化时校验版本，如果不匹配就清空重建。当前项目没有做这个，因为数据模型还比较简单，没有迁移需求。

#### Q18：多个模型切换时，正在流式输出中切换会怎样？

当前实现没有在模型切换时做特殊处理。如果用户在 AI 正在流式输出时切换了模型（通过 Business Store 的 `systemModelName`），正在进行的流不会中断，因为它绑定的是切换前的模型配置。下次发送消息时才会用新模型。

如果需要在切换模型时自动中断当前流，可以在 Business Store 的 `systemModelName` setter 里加一个 `messageStore.abortActiveStream()` 调用。这是一个可以优化的边界情况。

### 主题八：架构思考与改进方向

#### Q19：如果要加情绪分析功能，你会怎么设计？

情绪分析功能需要在对话结束后对整个会话内容做分析，生成结构化报告。设计上分三层：数据层、分析层、展示层。

数据层需要扩展 `chat.d.ts` 的类型定义，新增 `EmotionReport` 接口，包含情绪标签、评分、风险等级、关键词、建议等字段。在 Message Store 里新增 `reportsBySession: Record<string, EmotionReport>` 存储每个会话的报告。持久化复用现有的 localStorage 机制。

分析层在对话结束后（消息 status 变为 completed 时）触发。调用 LLM API 发送一个 system prompt，要求模型以结构化 JSON 格式输出情绪分析结果。返回的 JSON 需要做字段规范化——因为 LLM 输出不保证格式完全正确，要对缺失字段做默认值兜底，对评分做范围约束（0-10），对标签做枚举校验。

展示层用 ECharts 做数据看板：情绪趋势折线图（按时间轴）、情绪分布饼图、关键词词云排行。这些图表组件放在 `src/components/EmotionDashboard/` 目录下。路由层新增 `/dashboard` 路由，用 Vue Router 的路由守卫控制访问权限。

#### Q20：这个项目有什么架构短板？你会怎么改进？

主要有三个短板。第一是 API Key 安全：当前 Key 通过 `import.meta.env.VITE_*` 注入前端代码，构建后暴露在 JS bundle 里。生产环境必须改为 Node.js 代理服务，Key 留在服务端环境变量中。

第二是状态持久化方案：手动 localStorage 在数据量大时有容量和性能问题。中期应该迁移到 IndexedDB，可以用 `idb-keyval` 或 `Dexie.js` 封装。长期考虑加后端数据库做多端同步。

第三是测试覆盖：当前项目没有任何单元测试和集成测试。核心的流式解析逻辑（`splitStream`、`createParser`、`transformStreamValue`）非常适合用 Vitest 做单元测试——构造 mock 的 SSE 数据，验证解析结果。Store 层可以用 `@pinia/testing` 做集成测试。组件层可以用 Vue Test Utils 测试交互逻辑。这三个层面的测试补上之后，重构和加新功能的信心会大很多。
